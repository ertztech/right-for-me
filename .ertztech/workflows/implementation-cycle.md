# Implementation Cycle Workflow

Purpose:
Coordinate a focused one-to-two-day delivery period that may contain multiple independent implementations, each prepared and reviewed separately.

Status:
Preferred delivery container for NextMove.

Referenced workflows:
- ./parking-lot-review.md
- ./backlog-review.md
- ./backlog-refinement.md
- ./work-selection.md
- ./implementation-planning.md
- ./implementation-review.md
- ./implementation-retro.md

Cycle Rules:
- Default duration is one day.
- Maximum intended duration is two days.
- One ChatGPT conversation is preferred when practical.
- The normal Cycle Backlog contains one to three items.
- The Cycle Backlog is a forecast, not a commitment.
- Only one implementation is active at a time.
- Codex receives one approved Implementation Package at a time.
- The founder may stop, reorder, replace, remove, or defer work.
- Each implementation keeps its own Implementation Package, GitHub issue, Codex prompt, branch, commit, pull request, acceptance criteria, test checklist, Implementation Review, and merge decision.

Workflow Boundaries:
- Implementation Cycle coordinates the one-to-two-day delivery period.
- Parking Lot Review evaluates deferred ideas.
- Backlog Review prioritizes candidate work.
- Backlog Refinement is optional and readiness-driven.
- Work Selection may help choose the next item when priority is unclear.
- Implementation Planning prepares one implementation.
- Implementation Review evaluates one implementation.
- Implementation Retro reviews the entire cycle.
- Existing workflows remain authoritative for their detailed process.

Flow:
1. Open the cycle.
2. Review Cycle Status.
3. Run Parking Lot Review.
4. Run Backlog Review.
5. Run Backlog Refinement only when selected work is not ready.
6. Define the Cycle Goal.
7. Create the Cycle Backlog.
8. Select one implementation.
9. Run Implementation Planning.
10. Obtain Founder Approval.
11. Send one approved Implementation Package to Codex.
12. Run Implementation Review.
13. Complete the PR and merge, rework, or defer decision.
14. Update Cycle Status and the Cycle Backlog.
15. Repeat the implementation loop as appropriate.
16. Run one Implementation Retro.
17. Close the cycle.

Required Output:
1. Cycle goal
2. Cycle Status
3. Cycle Backlog
4. Current implementation decision
5. Founder decisions needed
6. Deferred, blocked, or removed items
7. Retro trigger and close recommendation
